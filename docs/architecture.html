Architecture // DietReport

Technical reference: package structure, data layer, reminder scheduling, error handling, and testing conventions.

# overview

DietReport is a fully offline Android app. No backend, no network calls, no user accounts. All data — meal logs, photos, settings — lives on the device.

# navigation

Single NavHost in MainActivity. Routes are string constants in the Routes object. All screens are Compose destinations.

Primary flow

HOME ──tap slot──→ LOG_MEAL/{slotId} ──confirm──→ LOCK/{slotId} ──done──→ HOME

First-launch onboarding

isOnboardingComplete = false ──→ SETTINGS ──first save──→ HOME

Notification deep-link

The "Log meal" notification action carries a slot_id extra. MainActivity.onCreate() reads it and navigates directly to LOG_MEAL/{slotId}, bypassing Home.

Bottom navigation tabs: HOME ↔ GALLERY ↔ SETTINGS ↔ REPORT. The bottom bar is hidden on the LOCK route.

# scheduling

⚡

SlotComputer.computeSlots() is a pure function. It takes wake time, first-meal delay, and bedtime in epoch millis and returns a list of SlotTime objects — no Android dependencies, fully unit-testable.

ReminderScheduler wraps AlarmManager.setExactAndAllowWhileIdle() and writes a ReminderSlot row to Room for each alarm so the Home screen can display upcoming slots without querying the AlarmManager.

What triggers rescheduling

1
SettingsViewModel On first save and every subsequent save. Rebuilds today's slots immediately.
2
BootReceiver Fired on ACTION_BOOT_COMPLETED. Exact alarms don't survive reboots — this restores them.
3
DailyRescheduleReceiver A midnight alarm. Builds all slots for the next day in advance.
4
TimezoneChangeReceiver Fired on ACTION_TIMEZONE_CHANGED. Cancels and rebuilds all upcoming slots in the new local time.

# errors

All app-level errors are modeled as sealed subclasses of AppError. The current subclass is SchedulerError(message: String) — extend it as needed; never swallow exceptions silently.

SchedulerErrorBus is an application-scoped SharedFlow. MainActivity.onResume() checks exact-alarm and notification permission status on every resume and posts SchedulerError to the bus when either is revoked. The Home screen collects the bus and surfaces a warning banner.

—

The bus pattern decouples the permission check (in Activity) from the UI response (in ViewModel/screen) without the two needing a direct reference to each other.

Pure logic classes (SlotComputer, SettingsValidator, ReportMetrics) are covered by unit tests — they have no Android dependencies so they run on the JVM.

Screen flows are covered by instrumented tests that launch MainActivity and drive the UI with UI Automator selectors via testTagsAsResourceId = true.
